<objective>
Establish the structural enforcement that makes SCOR-03 ("pure Dart, zero Flutter/IO dependencies in lib/domain/") a verifiable invariant rather than an aspirational comment. Every subsequent plan that adds files under `lib/domain/` will be gated by this test — a contributor who accidentally types `import 'package:flutter/material.dart';` inside a domain file gets a red `dart test` immediately.

Per RESEARCH.md §Test Strategy → Structural test, this is implemented as a single ~30-line Dart test using `dart:io`'s file APIs to walk `lib/domain/` and grep for forbidden imports. No `custom_lint` / `import_lint` analyzer plugin is needed for one rule (RESEARCH.md considered and rejected those as overkill for Phase 1).

Also formalize `dart_test.yaml` for consistent test output across local + future CI invocations.

Purpose: Enforce the pure-Dart domain boundary that ALL downstream phases depend on. Without this, Phase 2 might silently `import 'package:drift/drift.dart';` from a domain model, breaking testability and architectural cleanliness.

Output: A structural test that runs in <1s as part of `dart test`, plus a minimal `dart_test.yaml` for test runner defaults.
</objective>
